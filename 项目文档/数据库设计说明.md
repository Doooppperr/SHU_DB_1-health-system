# 康康健健 HealthDoc 数据库设计说明

> 文档状态：SQLite/openGauss schema v3 当前物理设计与迁移说明，更新于 2026-07-16。

## 1. 运行方式与版本

本地数据库：

```text
backend/instance/health_system.db
```

SQLite 使用 `PRAGMA user_version=3`。应用对新空库创建完整 v3；对非空旧库先检查版本和表结构，版本或约束不完整时拒绝启动，避免 `create_all()` 形成新旧表混合的半升级状态。

生产通过 `DATABASE_URL=opengauss+psycopg2://...` 连接 openGauss。非 SQLite 的 `create_all()` 只能创建缺失表，不能升级旧列和约束；需要用 `scripts/migrate_sqlite_to_gaussdb.py --replace` 将已经验证的当前 SQLite 快照作为完整数据源，重建 schema、复制全部表、重置生成序列并逐表核对行数。

## 2. 当前表结构

| 表 | 用途 | 关键约束 |
|---|---|---|
| `users` | 三类账号与普通用户健康身份 | 用户名/邮箱/健康身份码唯一，角色-机构-健康身份检查，启用状态 |
| `friend_relations` | 亲友关系与授权 | 用户对唯一，禁止自己添加自己 |
| `institutions` | 体检机构 | 名称与分院联合唯一，必填文本，软停用 |
| `packages` | 机构套餐 | 机构与名称联合唯一，价格非负，性别范围检查，软停用 |
| `institution_invites` | 每机构当前邀请码 | 机构唯一、哈希唯一、状态检查 |
| `institution_images` | 机构相册 | 存储键唯一，机构内排序唯一且为 0–7 |
| `indicator_categories` | 指标分类 | 名称唯一 |
| `indicator_dicts` | 指标标准字典 | 编码唯一，分类与名称联合唯一，参考范围检查，自测白名单标志 |
| `self_measurements` | 普通用户日常测量 | 用户/指标外键，值非负，测量时间索引 |
| `exam_registrations` | 用户体检登记 | 状态检查；用户+日期对非取消行部分唯一 |
| `institution_reports` | 机构正式报告及状态机 | 机构+健康身份码+日期对活动报告部分唯一，状态检查 |
| `report_indicators` | 报告标准化指标 | 报告+字典联合唯一，值非空，来源检查 |
| `comments` | 机构评论 | 评分 1–5，内容非空 |

AI 会话、AI 分析结果、验证码、JWT 和机构健康访问日志不建业务表。AI 会话只保留在当前浏览器标签页；验证码与令牌使用运行时机制。

## 3. 当前关系

```text
institutions 1 ── N users（institution_admin.managed_institution_id）
institutions 1 ── 0..1 institution_invites（当前邀请码）
institutions 1 ── N institution_images
institutions 1 ── N packages
institutions 1 ── N exam_registrations
institutions 1 ── N institution_reports
institutions 1 ── N comments

users 1 ── N self_measurements
users 1 ── N exam_registrations
users 1 ── N institution_reports（matched_user，可空）
users 1 ── N institution_reports（creator，可空，删除后保留快照）
users 1 ── N comments / friend_relations

exam_registrations 0..1 ── 0..1 institution_reports（匹配后双向引用）
institution_reports 1 ── N report_indicators
indicator_dicts 1 ── N self_measurements
indicator_dicts 1 ── N report_indicators
```

`exam_registrations.matched_report_id` 与 `institution_reports.exam_registration_id` 都是唯一可空外键，用于快速双向访问。匹配服务在同一事务内设置两端，避免单边悬挂。

## 4. 关键字段与约束

### 4.1 `users`

- `role`：`user`、`institution_admin`、`admin`。
- `managed_institution_id`：机构账号必填，其他角色必须为空；同机构可有多个账号。
- `health_id`：普通用户必填且全局唯一，管理员角色必须为空。
- `real_name`、`birth_date`、`gender`、`allergy_history`、`medical_history`：普通用户资料。
- `gender`：可空，非空时为 `male/female/other/undisclosed`。
- `is_active`：账号停用状态，默认真。
- `password_hash`：bcrypt 哈希，不保存明文。

### 4.2 `institutions`、`packages` 与 `institution_images`

- 机构和套餐通过 `is_active` 软停用。
- `(institutions.name, branch_name)` 唯一。
- `(packages.institution_id, name)` 唯一；`price >= 0`。
- 机构图片保存 `storage_key/image_url/sort_order`，机构内排序唯一且限制 0–7；最小排序是封面。
- 文件二进制保存在 `backend/uploads`，数据库只保存定位信息。

### 4.3 `institution_invites`

- `institution_id`：每机构唯一，表示“当前邀请码槽位”。
- `code_hash`：高熵邀请码哈希，全局唯一；API 永不返回哈希。
- `status`：`active/used/superseded`。
- `issued_by_admin_id`、`used_by_user_id` 删除账号时置空。
- `issued_at`、`used_at` 记录当前码生命周期。
- 重新签发更新同一行，旧码哈希被替换，因而旧明文立即失效。

### 4.4 `indicator_dicts`

- `code` 全局唯一，`(category_id, name)` 联合唯一。
- `aliases` 保存同一标准指标的 JSON 别名集合。
- `unit`、`reference_low/high`、`clinical_significance` 和 `value_type` 定义标准含义。
- `allow_self_measurement` 控制日常测量白名单；当前只有 `HEIGHT/WEIGHT/HR/TEMP/SPO2/FBG` 为真。

### 4.5 `self_measurements`

- `user_id` 删除时级联；只能指向普通用户（业务层验证）。
- `indicator_dict_id` 必须对应允许自测的字典项（业务层验证）。
- `value` 为 `Numeric(14,4)` 且非负。
- `measured_at` 保存实际测量时间；`created_at/updated_at` 保存系统时间。
- 同用户、同指标、同日不设唯一约束，允许多次测量。

### 4.6 `exam_registrations`

- `status`：`awaiting_report/matched/cancelled`。
- `user_id` 删除时级联；机构必填，套餐可空且删除套餐时置空。
- `exam_date` 是精确匹配字段。
- `matched_report_id` 唯一可空，报告删除时置空。
- 部分唯一索引 `uq_exam_registrations_active_user_date`：同一用户与日期仅允许一条 `status <> cancelled` 的登记。
- 已取消行保留审计时间 `cancelled_at`；用户可以在同日重新登记。

### 4.7 `institution_reports`

- `institution_id`：报告生产机构，不可由机构端请求切换。
- `created_by_user_id`：创建账号，删除时置空。
- `created_by_username_snapshot`：创建时用户名快照，即使账号删除仍保留。
- `subject_name_snapshot`、`subject_health_id`、`exam_date`：精确匹配事实。
- `package_id` 可空，套餐删除时置空。
- `matched_user_id` 可空，普通用户删除时报告级联删除。
- `exam_registration_id` 唯一可空，登记删除/取消关联时置空。
- `status`：`draft/locked/waiting_match/published/withdrawn/expired`。
- `ocr_diagnostics`：OCR 候选、未匹配、过滤与完整性诊断，不保存完整云响应。
- `temporary_file_url`：仅草稿期临时文件；锁定时清空并删除文件。
- `locked_at/submitted_at/expires_at/published_at/withdrawn_at`：状态时间。
- 部分唯一索引 `uq_institution_reports_active_subject_date`：同机构、同健康身份码、同体检日期只允许一个未撤下/未过期报告。
- `display_id=report{id}` 为派生属性，不是数据库列。

### 4.8 `report_indicators`

- `(report_id, indicator_dict_id)` 唯一，一个报告每个标准指标最多一行。
- `value` 为规范字符串且不能为空，可兼容字典定义的 numeric/text。
- `is_abnormal` 根据字典参考范围计算。
- `input_source`：`manual/ocr`。
- 报告删除时指标级联删除。

### 4.9 `friend_relations` 与 `comments`

- 亲友 `(user_id, friend_user_id)` 唯一且禁止自关联；`auth_status` 表示被添加方当前授权。
- 评论引用用户与机构，评分 1–5，`is_visible` 由管理员审核。
- “必须拥有该机构已发布报告才能评论”是跨表业务规则，由事务查询保证，不复制报告 ID 到评论表。

## 5. 完整性与事务规则

- 注册普通用户时生成唯一健康身份码；机构邀请消费、账号创建和绑定原子提交。
- 账号角色、机构绑定、健康身份和启用状态由数据库约束与逐请求业务校验共同保证。
- 套餐必须属于登记或报告的机构。
- 报告只有 `draft` 可写；锁定使用条件更新，之后的写请求返回 409。
- 精确匹配同时核对姓名、健康身份、日期和机构，并锁定/条件更新登记与报告。
- 双向匹配同时写入 `matched_user_id`、`exam_registration_id` 和 `matched_report_id`。
- 60 天到期由 `expires_at` 判定；到期报告不能再次匹配。
- 撤下只改变报告状态和时间；用户读取、趋势与 AI 统一过滤 `published`。
- 删除普通用户前显式确认，由外键级联清理自测、登记、匹配报告及报告指标。
- 删除机构账号时 `created_by_user_id` 置空，快照保留。
- 图片/临时文件保存失败或事务失败时补偿清理；锁定成功后删除临时报告文件。

## 6. 文件访问边界

- `uploads/institutions/`：机构图片；公开路由仍要核对数据库 `storage_key`。
- 草稿报告临时文件：只供机构草稿处理内部使用，不通过公共 `/uploads` 或用户 API 返回。
- `/uploads/reports/...` 明确返回 404。
- 报告锁定时删除临时文件；等待匹配、已发布和已撤下报告不保存原文件。
- `cleanup_local_runtime.py` 只保留当前草稿 `temporary_file_url` 与机构图片引用，其余视为孤儿；默认 dry-run。

## 7. schema v3 本地迁移机制

`backend/scripts/upgrade_local_database.py` 的步骤：

1. 解析目标文件并执行源库 `integrity_check`；
2. 检查当前版本、表、列和命名约束；已是 v3 时只验证并幂等退出；
3. 从现有 `users` 选择当前系统管理员，读取 `id/username/password_hash` 及可兼容字段；
4. 在同目录临时数据库按当前 SQLAlchemy metadata 建表；
5. 插入管理员，设置 `PRAGMA user_version=3`；
6. 检查结构、外键和完整性；
7. 复制原数据库为 `health_system.before-schema-v3-<时间戳>-<随机串>.db`；
8. 使用 `os.replace` 原子替换正式库；失败时删除临时文件并保持源库。

首次启动 v3 后，由种子逻辑补充机构、套餐、指标字典和本地演示账号/数据。

## 9. openGauss 全量演示快照同步

当前服务器用于演示，Git 中的合成 SQLite 快照是本次全量同步的权威数据源。同步流程必须：

1. 停止写入并对现有 openGauss、上传目录和环境配置生成恢复点；
2. 从 root-only 环境文件读取目标连接字符串，不在命令、日志或文档显示密码；
3. 执行 `migrate_sqlite_to_gaussdb.py --replace`，明确允许清空应用表并导入合成快照；
4. 验证所有表行数、生成序列、角色登录、健康检查和关键 API；
5. 删除服务器临时 SQLite 文件，避免留下重复账号数据副本。

如果服务器转为承载真实业务数据，不得继续使用 `--replace`。届时必须另行设计保留真实数据的增量迁移和回滚方案。
