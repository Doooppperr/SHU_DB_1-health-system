# 康康健健 HealthDoc 数据库设计说明

> 文档状态：融合 1.0—3.0 数据模型演进的 SQLite/openGauss 设计说明。当前物理基线为 schema v8，更新于 2026-07-21。

## 1. 设计目标

数据库服务于“用户健康旅程 + 机构履约服务 + 平台治理”，而不是把数据表直接暴露为产品模块：

- 用户从健康总览记录自主测量，在健康时间线查看自主记录与体检结果；
- 体检套餐是可版本化的服务产品，包含适用人群和健康领域；
- 一次同行体检安排可包含多名参与者，并保留完整状态事件；
- 机构按到检、录入、复核、归档的任务流生产健康档案；
- 满额候补、空位通知、文本结论和附件均是正式业务对象。
- 机构主体与分院分离：报告归档后可在同主体内跨院只读，运营数据仍归具体分院。

本地数据库位于 `backend/instance/health_system.db`，使用 `PRAGMA user_version=8`。生产环境通过 `DATABASE_URL` 连接 openGauss。

### 1.1 从 1.0 到 3.0 的数据模型演进

| 产品阶段 | 数据建模重点 | 结构结果 |
|---|---|---|
| 1.0 | 账号、机构、套餐、单人预约和基础健康记录 | 建立核心实体与角色外键，支撑最小业务闭环 |
| 2.0 | 自主测量、机构报告状态机、亲友授权和时间线 | schema v6 将个人记录与机构报告分开，报告形成草稿、锁定、归档的生命周期 |
| 3.0 第一阶段 | 健康领域、套餐版本、多人预约、容量候补、通知、文本结论和附件 | schema v7 将服务产品、历史快照、可靠通知和图文健康档案正式建模 |
| 3.0 分院阶段 | 机构主体、分院归属和跨院归档访问审计 | schema v8 新增 `organizations` 与 `report_access_logs`，报告同主体跨院只读，运营数据继续按分院隔离 |

迁移始终保留账号身份和已发生的历史事实。新结构通过迁移脚本显式升级，不能依靠 `db.create_all()` 猜测修改已有数据库。

## 2. 业务域与数据表

| 业务域 | 主要表 | 作用 |
|---|---|---|
| 账号与关系 | `users`、`friend_relations` | 三角色账号、普通用户健康身份、亲友只读授权 |
| 机构服务 | `organizations`、`institutions`、`institution_invites`、`institution_images`、`comments` | 机构主体、分院资料、工作人员准入、相册与服务评价 |
| 健康领域 | `indicator_categories`、`indicator_dicts`、`health_domains`、`indicator_domain_links` | 标准指标、可自主测量范围及领域归属 |
| 套餐产品 | `packages`、`package_versions`、`package_version_domains`、`package_change_requests` | 套餐身份、历史版本、适用领域和审核记录 |
| 预约服务 | `booking_groups`、`appointments`、`appointment_events`、`appointment_capacity_slots` | 同行安排、参与者预约、进度事件与可约名额 |
| 候补通知 | `waitlist_subscriptions`、`waitlist_subscription_participants`、`availability_notification_events`、`notification_outbox`、`notification_deliveries` | 满额候补、同行参与者、通知事实、可靠投递与尝试记录 |
| 健康档案 | `self_measurements`、`institution_reports`、`report_indicators`、`report_text_results`、`report_assets`、`report_asset_annotations`、`report_access_logs` | 自主测量、机构归档、数值指标、文本结论、附件、标注与跨院访问审计 |

AI 会话、验证码和 JWT 不进入业务数据库；AI 只读取获准的健康事实，不形成新的医疗档案。

## 3. 关键关系

```text
organization ── institution(branch) ── package ── package_version ── health_domain
      │             │
      │             └── booking_group ── appointment ── appointment_event
      │                                      │
      └──────────────── institution_report ──┼── report_indicator
                                             ├── report_text_result
                                             └── report_asset

user ── self_measurement
user ── appointment
user ── friend_relation（只读授权）
```

- `packages` 表示可持续识别的套餐产品，`package_versions` 保存每次获批后的名称、价格、说明、适用人群和生效事实。
- 预约保存所选套餐版本与展示快照，后续调价或下架不会改写历史安排。
- `booking_groups` 代表一次同行安排，`appointments` 代表其中每位受检者的服务记录。
- `appointment_events` 记录创建、取消、到检、归档等节点，时间线直接读取业务事件。
- 归档报告可同时包含结构化指标、文本结论与附件，不再把现实报告简化为单一数值表。
- 报告始终记录源分院；同机构兄弟分院只能读取 `published` 报告，详情和附件访问分别写入 `report_access_logs`。

## 4. 核心完整性规则

### 4.1 账号与权限

- `role` 仅允许 `user/institution_admin/admin`。
- 普通用户必须有唯一健康身份码且不能绑定分院；机构账号必须绑定具体分院；管理员两者均为空。
- 分院有业务数据后不得随意迁移机构主体；停用机构主体或分院只阻止新业务，不删除历史报告。
- 密码仅保存 bcrypt 哈希；停用账号不能登录、刷新令牌或继续受保护操作。
- 亲友授权只提供脱敏健康视图，不授予代记测量、修改资料或编辑报告能力。

### 4.2 套餐与预约

- 套餐业务身份与版本内容分离，预约引用明确版本并保存必要历史快照。
- 机构提交套餐变更，管理员审核通过后才产生或启用正式版本。
- 同行安排最多五名参与者；服务端统一复核每位参与者的日期冲突、套餐适用性和机构容量。
- `appointment_capacity_slots` 保存机构日期名额，候补订阅只在满额场景创建。
- 取消或失效释放名额，通知事件和 outbox 保证空位提醒可追踪、可重试且不重复发送。

### 4.3 健康档案

- 自主测量仅允许指标字典中开放的指标，同一指标同一天可以多次记录。
- 机构报告必须来自可履约预约；受检者、日期、机构和套餐版本来自预约事实。
- 报告归档后内容不可修改或撤下；数值指标、文本结论和附件随档案一起保留。
- 同一天同一指标存在机构结果时，趋势优先使用机构值；否则采用当天最后一次自主测量。
- 亲友、趋势、时间线和健康 AI 读取同一套权限过滤后的归档事实。
- “体检数据”列表只查询机构归档报告；个人测量继续由总览、时间线和趋势读模型读取。

## 5. 快照与 JSON 的使用边界

以下字段是发生当时的业务事实，而非重复缓存：

- 预约中的受检者、套餐版本、名称、价格和联系方式快照；
- 报告中的受检者与创建者展示快照；
- 套餐审核的变更前后内容；
- 报告识别诊断、通知载荷和外部投递结果。

实体关系、指标明细、套餐领域和同行参与者仍使用关系表表达。JSON 只保存审计快照或外部交互载荷，不替代可查询的业务关系。

## 6. 事务与并发

- 邀请码消费与机构账号创建在同一事务完成。
- 同行体检安排先校验全部参与者与名额，再整体提交；任一参与者不满足条件时全部回滚。
- 机构容量变更、预约创建/取消、候补触发在锁定的日期容量记录上处理。
- 报告归档与预约履约状态同步提交，避免用户看到“已出报告但预约未完成”的不一致状态。
- 套餐审核通过与正式版本生效同步提交。
- 通知采用 outbox：业务事务只写待投递事件，发送失败不回滚已成立的预约事实。

## 7. 迁移与演示数据重置

检查与升级本地结构：

```powershell
Set-Location .\backend
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py --check-only
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py
```

覆盖旧演示记录并保留原测试账号：

```powershell
.\.venv\Scripts\python.exe .\scripts\reset_v8_demo_data.py --check-only
.\.venv\Scripts\python.exe .\scripts\reset_v8_demo_data.py --apply --yes
```

重置脚本仅面向本地合成演示库；发现非演示个人或机构账号时会拒绝执行，不得用于生产数据库。
