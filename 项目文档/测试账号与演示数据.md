# 康康健健 HealthDoc 3.0 测试账号与演示场景

> 更新日期：2026-07-20。本文及本地 SQLite 快照全部为合成演示数据，不对应任何真实个人、医疗机构或工作人员。

## 1. 使用原则

- 演示以“现实人物完成一段健康服务旅程”为目标，不再为每个账号机械复制全部数据库状态。
- `backend/instance/health_system.db` 是可直接启动的 schema v7 合成快照；不得写入真实健康资料后提交。
- 本文、合成 SQLite 和 9 个水印附件均随 GitHub `main` 保留；不得从提交中删除测试账号说明或只上传代码而遗漏演示快照。
- 正常启动不会覆盖本地操作。只有显式运行受保护的重置脚本，才会重建演示业务数据。
- 套餐表示健康领域服务，不承诺固定指标清单；机构实际完成的合法结果决定最终健康数据内容。
- 空位提醒不占位、不自动预约；代预约权限也不包含健康数据查看权限。

## 4. 普通用户演示旅程

### `test1`：完整主旅程

1. 登录后在健康总览查看今日个人记录、下一次体检、最新机构健康数据和空位提醒。
2. 点击“记录今日测量”，新增、编辑、删除一条测量；完整历史从健康数据进入。
3. 在健康时间线切换“全部、体检记录、个人记录”。个人测量按自然日合并；综合记录超过 15 条，可验证翻页。
4. 打开已完成的心电与循环检查，查看结构化指标、机构文字结论、带演示水印的检查影像和批注。
5. 查看自己、`test2`、`test3` 的三人家庭预约：`test2` 同时授权查看和代预约，`test3` 只授权代预约。
6. 查看已发送过提醒的空位订阅，确认系统明确提示“提醒不代表预约成功，也不会保留名额”。

### `test2`：代谢随访与来源对照

- 连续个人记录和两家机构的糖脂代谢健康数据。
- 健康趋势按来源分轨，同日机构和个人记录不会相互覆盖。
- 已向 `test1` 授予健康查看和代预约两种独立权限。

### `test3`：长辈代预约权限边界

- 具有心脑血管和肾脏关注的历史健康数据。
- `test1` 可以把 `test3` 加入预约，但不能切换查看其健康数据、趋势或检查影像。

### `test4`：改约、图片结果与评价

- 存在一次真实取消记录和后续重新预约。
- 已完成肝胆代谢检查，包含检查图片和机构批注。
- 存在一条待管理员审核的机构评价。

### `test5`：多机构来源隔离

- 同一业务日期来自不同机构的结果分别保留，不合并、不互相覆盖。
- 具有呼吸专项健康数据和一条过去未到检的失效预约。
- 未授权亲友不能读取其健康资料。

## 5. 机构与管理员旅程

使用 `institution1_staff1`：

1. 在运营总览查看今日容量、待到检、待归档、已完成和空位提醒。
2. 从今日接待确认到检，再进入待归档任务。
3. 按“指标—文字结论—检查附件—复核—提交归档”完成健康数据生产。
4. 查看多人预约组，但只在鉴权后的详情中读取必要受检者信息。
5. 查看当前套餐、历史版本、待审核变更和管理员反馈。

使用 `demo_admin`：

1. 查看一条待审核、一条已通过和一条有明确理由的驳回记录。
2. 审核待公开评价。
3. 确认管理员只能治理账号、机构和公开内容，不能读取健康数据正文或检查附件。

## 6. 安全重置演示数据

停止后端后，在 `backend` 目录先预检：

```powershell
.\.venv\Scripts\python.exe .\scripts\reset_v7_demo_data.py --check-only
```

确认目标是本地 schema v7 合成库、账号清单完全匹配后，再按脚本帮助信息提供显式确认参数执行。脚本会：

1. 备份 SQLite 和演示附件；
2. 保留账号认证数据与机构账号绑定；
3. 在事务中替换机构资料、套餐、预约、健康数据、个人记录、授权、候补、通知、评价和审核记录；
4. 生成不含真实隐私的演示检查图片；
5. 校验账号快照、外键、数据库完整性和附件 SHA-256；
6. 确认旧 A/B/C/D/E 套餐及 `BG-LEGACY` 预约组已经清除。

发现陌生账号、非 SQLite 数据库、非 schema v7 或不完整机构绑定时，脚本必须拒绝执行。应用正常启动绝不会自动清空已存在的业务记录。

## 7. 验收标准

- 登录 `test1` 即可看到完整、有上下文的用户故事，而不是平均分布的状态矩阵。
- 侧栏没有独立“日常测量”；旧地址会返回健康总览并打开测量面板。
- 时间线包含按天聚合的个人记录和一次体检一张卡的体检记录。
- 机构详情不显示 `all`、`female_all`、数据库 ID 或固定指标合同。
- 套餐、预约、候补、健康数据和机构任务均使用现实业务文案。
- AI 入口不遮挡列表、操作按钮或移动端内容。
- 后端测试、前端测试、生产构建、SQLite `integrity_check` 和外键检查全部通过。
