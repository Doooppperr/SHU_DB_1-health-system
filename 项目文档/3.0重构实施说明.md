# HealthDoc 3.0 重构实施说明

> 当前状态：schema v8 已实施，在第三轮平台化界面上完成机构主体/分院协作及全新现实场景演示快照，日期 2026-07-20。

## 已完成能力

- 用户端拆分为健康时间线、体检数据和健康趋势。时间线统一分页展示“一次体检一张卡”和“个人记录按自然日聚合”；“体检数据”只展示机构已归档报告，个人测量仅保留在总览、时间线和趋势来源轨道；健康趋势按领域返回多指标、来源分轨和附件事件。
- 日常测量不再作为独立导航模块，而是健康总览中的快捷任务；总览通过专用读模型展示今日记录、下一预约、最新健康数据、空位提醒和最近历程。
- 用户机构详情、套餐、预约、候补、健康数据列表和机构任务工作台采用现实业务卡片与步骤流。内部枚举、数据库 ID 和“原子提交”等实现术语不再直接展示给用户。
- AI 助手从可自由拖动的悬浮球改为顶栏入口；宽屏重排内容，窄屏覆盖显示，不再缩放整个应用或遮挡列表操作。
- 健康数据筛选区统一约束日期范围、下拉框和操作按钮的弹性宽度；Element Plus 日期范围组件不再越过所属列覆盖“资料来源”，宽屏与窄屏都沿用同一响应式规则。
- 平台健康领域、指标主/关联领域、套餐不可变版本和预约领域快照已经建模。手工录入、OCR、锁定、展示、趋势和 AI 共用 `services/domain_rules.py`。
- `POST /api/booking-groups` 原子创建 1–5 个预约名额，代预约与健康查看分别授权；机构预约接口支持日期、状态和分页看板。
- 空位提醒不占位、不自动预约。阈值变化写入通知事件与事务 Outbox，常驻 `scripts/notification_worker.py` 每 5 秒条件领取、发送、重试并记录投递，误开多个 worker 也不会重复发送；本地一键启动和服务器 systemd 均已集成。
- 健康数据支持领域文字结论、JPEG/PNG/WebP/PDF 私有附件和机构文字批注；草稿锁定后一起不可变。OCR 临时件与永久附件采用不同目录、模型和清理规则。
- AI 分析支持领域范围，结构化数据和图片同意分离；当前默认模型未声明图像能力时，图片分析接口明确禁用。
- schema v8 新增机构主体与分院层级。机构账号仍绑定具体分院：本院可生产报告，同机构兄弟分院仅可查看已归档报告及附件并留下跨院访问审计；草稿不共享，跨机构访问拒绝。预约、容量、候补、套餐、邀请码、评价和运营统计始终按分院隔离。
- 管理端采用“创建机构主体—添加分院—配置分院—签发分院邀请码”的流程；用户端以机构主体分组展示 15 家分院，地址、套餐、评价和预约入口均明确归属分院。

## 兼容与迁移

- 旧 `/api/exam-reports`、单人 `/api/appointments`、个人测量详情和单指标趋势接口暂时保留；新界面使用 v8 读模型和“体检数据”术语。
- SQLite 使用 `scripts/upgrade_local_database.py` 完成 v4–v7→v8 重建、公共列复制、完整性/外键/结构校验、备份和原子替换；v7 分院主键及业务外键原地保留，并建立对应机构主体。
- openGauss/GaussDB 使用 `migrations/versions/20260720_schema_v8.py` 增量迁移，已有生产库不依赖 `create_all()` 补列。
- `scripts/backup_recovery_set.py` 把 SQLite 在线快照、永久附件和 SHA-256 清单打包为同一恢复集合。

## 核心接口

- `GET /api/health/dashboard`
- `GET /api/health/timeline?record_type=all|exam|self`
- `GET /api/health-data`、`GET /api/health-data/{health_data_id}`
- `GET /api/health-trends/{domain_id}`
- `GET /api/organizations`、`GET /api/organizations/{id}`
- `GET /api/org/context`
- `GET /api/org/reports?scope=branch|organization`
- `POST /api/booking-groups`
- `POST/GET/DELETE /api/waitlist-subscriptions`
- `POST/PATCH/DELETE /api/org/health-data/{id}/assets`

## 验证

后端原有回归与 schema v8 分院权限测试、前端 Vitest、Vite 生产构建、SQLite v7→v8 迁移和正式本地数据库完整性校验均纳入本轮交付。准确数量以最后一次命令输出为准。

本地演示快照由 `scripts/reset_v8_demo_data.py` 显式重建。脚本只允许作用于预期的本地 schema v8 合成库，执行前备份数据库和附件，并逐字段确认原 13 个账号认证快照未变化；应用正常启动不会自动覆盖已有业务操作。

GitHub `main` 跟踪合成 SQLite、15 张分院封面、15 个带“演示数据/非真实医学影像”水印的报告附件及测试账号说明。服务器演示环境通过 `deploy-server.ps1 -SyncDemoDatabase` 在同一备份/回滚单元内同步代码、数据库和附件，并保留服务器既有 SMTP 配置；普通发布仍不会覆盖服务器数据。
