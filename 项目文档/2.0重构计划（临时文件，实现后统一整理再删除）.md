# HealthDoc 2.0 功能重构计划

> 文档性质：临时实施文档。
> 使用方式：作为 2.0 重构期间的需求、设计、实施和验收依据。
> 清理要求：重构完成并将最终内容统一整理进正式项目文档后，删除本文件。
> 编制日期：2026-07-16。

## 1. 重构目标

当前系统以健康档案表及其增删改查为中心，普通用户可以创建、OCR 上传、修改和删除体检报告及报告指标，而机构只能只读查看。这与真实业务中的数据生产责任相反。

2.0 版本将系统重构为以下业务模式：

- 机构负责生产体检报告和标准化体检指标。
- 普通用户只能读取已经与本人匹配的机构体检数据。
- 普通用户可以维护个人资料、登记体检事件，并录入本人日常自测数据。
- 用户通过统一健康时间线查看体检事件和日常测量。
- 同类指标汇入统一时间序列，机构数据优先于同日用户自测数据。
- 亲友在获得授权后，可以只读查看健康时间线、指标趋势，并在逐次同意后使用 AI 分析。
- 系统管理员只管理账号、机构、机构账号和邀请码，不接触健康内容。

AI 的安全边界、SSE 交互、公共知识 RAG 和确定性计算框架暂不重构，只适配新的健康数据来源和权限模型。

## 2. 最终业务边界

### 2.1 普通用户可以执行的操作

- 注册、登录和维护账号信息。
- 查看健康身份码。
- 维护真实姓名、出生日期、性别、过敏史和既往史。
- 浏览机构资料、套餐、相册和评论。
- 登记某日参加体检的机构，并可选择套餐或“暂不选择”。
- 在机构尚未匹配报告时查看“机构未提交”状态。
- 只读查看匹配后的标准化体检指标。
- 录入、修改和删除本人日常自测数据。
- 查看本人统一健康时间线和指标趋势。
- 管理亲友关系和访问授权。
- 在每次明确选择和同意后，使用本人或已授权亲友的健康数据进行 AI 分析。

普通用户不能执行以下操作：

- 上传体检报告或体检报告原文件。
- 新建、修改、删除机构体检报告。
- 新增、修改、删除机构报告指标。
- 修改体检报告的机构、套餐、体检日期或发布状态。
- 查看 OCR 原文件。
- 替亲友登记体检或维护日常测量。

### 2.2 机构账号可以执行的操作

- 维护本机构公开资料、套餐和相册。
- 使用 OCR 上传临时报告并检查映射结果。
- 手工选择标准指标并填写值。
- 在确认前修改临时录入内容。
- 确认并锁定报告内容。
- 提交锁定后的报告，触发精确匹配。
- 查看本机构创建的草稿、等待匹配、已匹配、已撤下报告。
- 撤下本机构报告。

机构账号不能执行以下操作：

- 浏览用户的体检登记待办。
- 搜索平台全部用户。
- 查看用户日常测量、健康时间线、其他机构报告、过敏史或既往史。
- 查看其他机构创建的报告。
- 修改已经确认锁定的报告或指标。
- 通过前端指定或伪造报告所属机构。
- 使用普通用户或亲友健康数据调用 AI。

一个机构可以绑定多个机构账号，所有机构账号权限相同，本轮不再细分录入员、审核员等机构内部角色。

### 2.3 系统管理员可以执行的操作

- 创建、修改、停用和恢复机构。
- 管理普通用户账号。
- 查看机构下绑定的机构账号。
- 为指定机构生成单次使用的邀请码。
- 删除机构账号。
- 查看不包含健康内容的账号和机构运行统计。

系统管理员不能执行以下操作：

- 查看体检报告、报告指标或 OCR 文件。
- 查看日常测量、健康时间线或指标趋势。
- 创建、修改、撤下或删除机构报告。
- 替机构上传或录入报告。
- 查看亲友健康数据。

现有管理员“档案监管”页面、管理员报告 CRUD 接口和健康内容统计全部移除。

## 3. 健康身份与个人资料

### 3.1 健康身份码

系统为每个普通用户生成全平台唯一的健康身份码，例如：

```text
HID-8F4K2M9Q
```

规则如下：

- 注册成功时由服务端生成，用户不自行输入。
- 使用高熵、不可顺序枚举的随机编码。
- 注册后不可修改。
- 用户可以在个人中心查看和复制。
- 用户实际体检时向机构提供真实姓名和健康身份码。
- 健康身份码不向亲友、评论接口或 AI 模型公开。
- 机构只能在报告录入流程中提交健康身份码，不能通过健康身份码搜索用户。

健康身份码是平台内的匹配标识，不称为身份证号或实名认证凭证。

### 3.2 用户健康资料

普通用户可以维护：

- 真实姓名。
- 出生日期；年龄由系统实时计算，不单独持久化。
- 性别。
- 过敏史，自由文本。
- 既往史，自由文本。

亲友不能查看上述个人资料。机构只在报告匹配时使用用户主动向机构提供的真实姓名和健康身份码，不通过平台读取完整个人资料。

已匹配报告保存提交时的受检者姓名快照。用户以后修改姓名，不反向改变历史报告。

## 4. 体检登记与机构报告流程

### 4.1 用户体检登记

用户从健康总览快捷按钮、侧栏“登记体检”或机构详情入口进入独立登记页，填写：

- 体检机构。
- 体检日期，默认当天。
- 体检套餐，可选；套餐下拉列表第一项为“暂不选择”。

约束如下：

- 同一用户、同一体检日期只能存在一条有效体检登记，不论机构。
- 未匹配前用户可以取消登记。
- 匹配成功后登记不可修改、取消或删除。
- 用户端未匹配状态统一显示“机构未提交”，卡片不可展开。
- 独立登记页必须提示：机构未匹配的体检数据只保留 60 天，超过期限后无法匹配。

### 4.2 机构报告录入和锁定

机构可以通过两种方式创建报告：

1. OCR 上传报告，自动映射标准指标后人工检查。
2. 手工选择标准指标并填写值。

本轮不建设套餐指标模板。套餐只作为可选业务信息，不决定报告必须填写的指标集合。

报告状态流转如下：

```text
draft
  -> locked
      -> waiting_match
          -> published
          -> withdrawn
          -> expired
      -> published
      -> withdrawn

published -> withdrawn
```

状态含义：

- `draft`：临时录入状态，允许 OCR、手工增删改指标和修改受检者匹配信息。
- `locked`：机构已经确认内容，报告及指标不可再修改，只能提交或撤下。
- `waiting_match`：已经提交，但平台尚未找到对应的有效用户体检登记。
- `published`：匹配成功，标准化指标已经向用户只读开放。
- `withdrawn`：机构主动撤下，不再向用户提供数据，也不再参与趋势聚合。
- `expired`：等待匹配超过 60 天，永久失去匹配资格，随后物理清理。

确认锁定时删除 OCR 原文件和临时 OCR 附件，只保留机构已经确认的标准化数据及必要的解析诊断摘要。

### 4.3 精确匹配规则

匹配使用以下四个条件：

- 受检者真实姓名。
- 健康身份码。
- 体检日期。
- 当前机构账号绑定的机构。

机构不能通过请求参数指定报告所属机构，机构 ID 必须从当前登录账号的服务端绑定关系取得。

匹配只允许返回：

- 匹配成功。
- 未找到对应体检登记。

接口不能返回候选用户、相似姓名、用户联系方式或其他健康信息。

匹配在两个方向触发：

1. 机构提交锁定报告时，查询是否已有对应体检登记。
2. 用户创建体检登记时，查询该机构是否已有未过期、未撤下的等待匹配报告。

匹配必须在单个数据库事务中完成，并通过唯一约束和条件更新避免一条登记匹配多份报告或一份报告匹配多条登记。

### 4.4 60 天保留规则

- 未找到体检登记的机构报告进入 `waiting_match`。
- `expires_at = submitted_at + 60 天`。
- 60 天内用户补充登记时允许自动匹配。
- 到达 `expires_at` 后立即视为失效，即使尚未完成物理删除也禁止匹配。
- 系统启动、机构报告查询、用户登记和匹配操作时执行过期清理。
- 提供独立清理命令，供部署脚本或计划任务主动执行。
- 过期清理删除报告、报告指标和残留临时文件。
- 已经匹配为 `published` 的报告不再受 60 天限制。

## 5. 日常健康数据

### 5.1 第一版可录入指标

- 身高。
- 体重。
- 心率。
- 体温。
- 血氧。
- 空腹血糖。

本轮暂不支持血压、步数、睡眠、设备同步和第三方数据来源。

### 5.2 日常测量字段

每条日常测量包含：

- 测量用户。
- 指标类型。
- 数值。
- 系统指标字典提供的固定单位。
- 测量时间。
- 创建时间。
- 修改时间。

用户选择指标后，单位由系统自动显示，前端不能自由填写单位。后端仍需校验指标是否属于允许自测的白名单。

用户一天可以录入同一指标多次，所有原始测量均保留，并可以单独修改或删除。

## 6. 统一健康时间线与指标趋势

### 6.1 健康时间线

健康时间线聚合以下事件：

- 用户日常测量。
- 用户体检登记。
- 已匹配的机构体检报告。
- 机构撤下事件。

示例：

```text
2026-07-15  自测体重 72.4 kg
2026-07-12  institution1 体检报告 · 机构已提交
2026-07-08  自测心率 76 次/分
2026-06-20  institution2 体检登记 · 机构未提交
```

用户只查看标准化体检数据，不查看 OCR 原文件或报告原文。

### 6.2 每日有效指标规则

趋势聚合以“用户 + 指标 + 日期”为粒度：

1. 当日存在未撤下的机构报告指标时，使用机构指标。
2. 机构报告没有该指标时，使用用户当日最后一次自测。
3. 当日没有机构指标时，使用用户当日最后一次自测。
4. 用户同日更早的测量仍然保留在历史列表中，但不进入默认趋势主线。
5. 机构报告撤下后，该日重新回退到用户当日最后一次自测。
6. 机构上传 B 指标、用户自测 A 指标时互不冲突，分别进入各自时间序列。

由于同一用户同一天只能登记一家机构，不处理同日多家机构的正式指标竞争问题。

### 6.3 趋势页面

趋势页面使用一个通用可视化区域：

- 选择本人或已授权亲友。
- 选择指标。
- 选择时间范围。
- 折线图展示每日有效值。
- 展示最新值、最高值、最低值和有效记录数。
- 图表标识数据来自机构报告还是用户自测。
- 点击数据点可以查看对应体检事件或日常测量详情。

不为每个指标开发独立页面或重复组件。

## 7. 亲友权限

现有亲友授权机制继续保留，由被查看方明确授权后生效。

授权亲友可以：

- 查看对方健康时间线。
- 查看对方标准化体检指标。
- 查看对方日常测量。
- 查看对方指标趋势。
- 在每次重新选择并明确同意后，将对方健康数据交给 AI 分析。

授权亲友不能：

- 查看真实姓名、出生日期、性别、健康身份码、联系方式、过敏史和既往史。
- 新增、修改或删除对方日常测量。
- 替对方登记或取消体检。
- 修改、撤下或删除对方报告。

亲友接口必须返回专用只读 DTO，不能复用包含个人资料的用户接口响应。

## 8. 机构账号与邀请码

### 8.1 多机构账号

- 删除 `users.managed_institution_id` 的唯一约束。
- 一个机构允许绑定多个 `institution_admin` 账号。
- 每个机构账号仍然只能绑定一个机构。
- 所有机构账号权限相同。
- 机构账号不能进入普通用户工作区。

### 8.2 邀请码

- 邀请码单次使用。
- 系统管理员为指定机构点击“生成下一枚邀请码”。
- 邀请码明文仅在生成响应中展示一次，数据库只保存哈希。
- 机构人员注册成功后，邀请码立即失效。
- 下次新增机构人员时，管理员重新生成下一枚。
- 不提供单独的重置和撤销操作。
- 生成下一枚时，如果存在尚未使用的旧码，则自动使旧码失效。

### 8.3 账号删除

系统管理员删除普通用户时，级联删除：

- 用户账号和个人资料。
- 日常测量。
- 体检登记。
- 已匹配机构报告和报告指标。
- 亲友关系。
- 评论及其他用户关联业务数据。

删除操作必须要求二次确认，并明确提示不可恢复。系统管理员无需也不能预览健康内容。

系统管理员删除机构账号时：

- 删除该登录账号，不降级为普通用户。
- 该账号历史创建的机构报告继续保留。
- 报告保存提交账号的用户名快照。
- 上传账号外键允许置空或使用等价的历史快照机制。

## 9. AI 适配

AI 主体能力暂不重构，只完成以下适配：

- 档案上下文改为标准化机构报告和日常测量。
- 明确标注每项数据属于机构报告还是用户自测。
- 趋势事实使用服务端计算的每日有效指标，不让模型自行决定同日数据优先级。
- 健康身份码、真实姓名和其他个人资料不传入模型。
- 亲友数据继续要求逐次选择和逐次同意。
- 私人健康数据仍不进入公共向量库。
- 管理员和机构账号仍不能提交私人健康数据给 AI。

## 10. 数据模型实施建议

### 10.1 `users`

新增或调整字段：

- `real_name`。
- `health_id`，唯一、不可空。
- `birth_date`。
- `gender`。
- `allergy_history`。
- `medical_history`。
- `is_active`。
- `managed_institution_id` 保留，但删除机构内唯一约束。

系统管理员账号不要求健康身份字段；机构账号只要求机构绑定；健康身份字段只对普通用户生效。应使用数据库检查约束保证角色与字段组合合法。

### 10.2 `self_measurements`

建议字段：

- `id`。
- `user_id`。
- `indicator_dict_id`。
- `value`。
- `measured_at`。
- `created_at`。
- `updated_at`。

复用现有标准指标字典，并通过字段或服务端白名单标记哪些指标允许用户自测。

### 10.3 `exam_registrations`

建议字段：

- `id`。
- `user_id`。
- `institution_id`。
- `package_id`，可空。
- `exam_date`。
- `status`：`awaiting_report/matched/cancelled`。
- `matched_report_id`，可空。
- `created_at`。
- `cancelled_at`，可空。

对未取消的有效记录建立“用户 + 体检日期”唯一约束或等价的事务校验。

### 10.4 `institution_reports`

建议字段：

- `id`。
- `institution_id`。
- `created_by_user_id`，允许机构账号删除后置空。
- `created_by_username_snapshot`。
- `subject_name_snapshot`。
- `subject_health_id`。
- `exam_date`。
- `package_id`，可空。
- `matched_user_id`，可空。
- `exam_registration_id`，可空且唯一。
- `status`。
- `ocr_diagnostics`，不保存原始文件和完整云端响应。
- `locked_at`。
- `submitted_at`。
- `expires_at`。
- `published_at`。
- `withdrawn_at`。
- `created_at`。

机构内对同一健康身份码和体检日期的有效报告建立唯一约束，防止同一机构重复提交。

### 10.5 `report_indicators`

建议字段：

- `id`。
- `report_id`。
- `indicator_dict_id`。
- `value`。
- `is_abnormal`。
- `input_source`：`manual/ocr`。

同一报告内同一标准指标唯一。所有写接口在报告离开 `draft` 后必须拒绝修改。

## 11. API 重构建议

### 11.1 普通用户 API

- `GET/PUT /api/profile/me`
- `GET/POST /api/self-measurements`
- `PUT/DELETE /api/self-measurements/<id>`
- `GET/POST /api/exam-registrations`
- `DELETE /api/exam-registrations/<id>`，仅允许取消未匹配登记
- `GET /api/exam-reports`
- `GET /api/exam-reports/<id>`
- `GET /api/health/timeline`
- `GET /api/health/trends/<indicator_dict_id>`

### 11.2 机构 API

- `GET/POST /api/org/reports`
- `GET /api/org/reports/<id>`
- `POST /api/org/reports/ocr`
- `POST/PUT/DELETE /api/org/reports/<id>/indicators`
- `POST /api/org/reports/<id>/lock`
- `POST /api/org/reports/<id>/submit`
- `POST /api/org/reports/<id>/withdraw`

机构报告接口始终从当前账号取得机构范围，不能接受跨机构查询条件。

### 11.3 系统管理员 API

保留账号、机构和邀请码接口，删除：

- 管理员报告列表和详情。
- 管理员创建、修改、删除报告。

## 16. 最终验收标准

- 普通用户无法通过界面或 API 创建、修改、删除机构报告和报告指标。
- 机构无法查看用户体检登记列表、其他机构报告和用户其他健康信息。
- 系统管理员无法获取任何报告、指标、测量或时间线内容。
- 报告确认锁定后，所有数据修改接口均被服务端拒绝。
- 机构先提交和用户先登记两种顺序均能自动匹配。
- 未匹配报告超过 60 天后永远不能再次匹配并最终完成清理。
- 同一用户同一天无法登记两家机构。
- 用户只查看标准化体检数据，不查看 OCR 原文件。
- 用户一天可以录入同一指标多次，趋势采用最后一次。
- 同日存在机构同类指标时趋势采用机构值，自测原始记录不丢失。
- 机构未提供的指标仍可使用用户自测值。
- 亲友只能在授权范围内只读访问，并可在逐次同意后使用 AI。
- 一个机构可以拥有多个同权限机构账号。
- 系统管理员删除普通用户时完整清理关联业务数据。
- 删除机构账号不破坏历史机构报告。
- 模拟账号、时间线、趋势、等待匹配、撤下和过期场景可以完整演示。
