# 康康健健 HealthDoc：项目需求与技术方案

> 文档状态：融合 1.0—3.0 的完整需求与技术基线。当前运行版本为 3.0，数据库为 schema v8，更新于 2026-07-21。

## 1. 项目定位

HealthDoc 是面向现实生活的个人健康与体检服务平台，不是把数据库表包装成增删改查页面。平台连接普通用户、体检机构与系统运营人员，围绕以下完整旅程设计：

1. 用户持续记录日常健康状态，并在同一时间线看到自主记录与正式体检结果；
2. 用户按服务能力、适用人群、健康领域和可约日期选择体检套餐，可为同行家人统一安排；
3. 机构从预约到检开始，完成报告录入、复核和归档；
4. 套餐以版本化服务产品管理，平台审核变更但不改写历史订单；
5. 亲友访问依赖明确授权，管理员治理平台但不读取个人健康正文；
6. AI 与报告识别仅提供辅助，不替代医疗判断和人工复核。

### 1.1 从 1.0 到 3.0 的产品演进

| 阶段 | 解决的核心问题 | 形成的能力 | 对当前版本的影响 |
|---|---|---|---|
| 1.0：基础业务闭环 | 让用户、机构和管理员能够在同一系统完成账号、机构浏览、预约和基础健康记录 | 建立三角色、基础预约、个人资料、机构资料和健康数据雏形 | 账号体系、角色入口和基础服务关系继续保留 |
| 2.0：健康档案与机构履约 | 把零散 CRUD 提升为“预约—到检—报告—归档”的可追踪流程 | 引入自主测量、健康时间线、趋势、机构报告草稿/锁定/归档、亲友只读授权和更清晰的角色权限 | 报告归档不可改写、亲友授权最小化和任务式机构工作台成为长期规则 |
| 3.0：现实平台与分院协作 | 让产品更接近真实体检服务平台，而不是数据库管理界面 | 引入健康领域、套餐版本、多人预约、容量候补、可靠邮件、图文报告、机构主体/分院、跨院归档只读和平台化 UI | schema v8 与本文件后续全部需求构成当前唯一实现基线 |

版本号描述产品能力阶段，schema 版本描述数据库结构，两者不是一一对应关系。3.0 先在 schema v7 完成健康领域、套餐和预约重构，再由 schema v8 增加机构主体与分院协作；当前开发、测试和部署均只面向 3.0/schema v8。

## 2. 页面体系与角色

系统采用“公开门户 + 三角色工作台”。

| 角色 | 默认入口 | 核心任务 |
|---|---|---|
| 访客 | `/` | 了解平台、登录注册、访客导览 |
| 普通用户 | `/dashboard` | 健康总览、快捷测量、时间线、趋势、体检安排、机构服务、亲友和健康 AI |
| 机构账号 | `/org/dashboard` | 预约到检、报告生产、机构资料、相册、套餐版本与审核申请 |
| 系统管理员 | `/admin/dashboard` | 机构、套餐审核、邀请码、账号状态和评论治理 |

三类登录角色具有独立导航、路由和后端接口。机构账号不能切换到其他机构；管理员不拥有个人测量、报告正文或健康 AI 上下文权限。

## 3. 业务需求

### 3.1 健康总览与自主测量

- 总览展示健康摘要、领域状态、最近体检、近期自主记录和待办提醒。
- 日常测量不设独立导航板块；总览保留“记录测量”按钮，以抽屉/对话框完成录入和修改。
- 支持身高、体重、心率、体温、血氧、空腹血糖等开放指标，同一指标同日可记录多次。
- 旧 `/measurements` 地址兼容返回总览，避免书签失效。
- 个人资料包含联系方式、真实姓名、生日、性别、过敏史和既往史；系统生成的健康身份码只读。

### 3.2 健康时间线与趋势

- 时间线统一承载体检记录和自主记录，可按 `all/exam/self` 筛选。
- 自主记录保留用户自己上传/录入的时间与来源；体检记录可展开结构化指标、文本结论和附件。
- 预约创建、取消、到检、报告归档等节点以业务事件呈现，不把内部状态码直接当作产品主界面。
- 趋势按健康领域和指标组织，支持本人及已授权亲友。
- 同日存在机构归档指标时优先使用机构值；机构未提供该项时采用当天最后一次自主测量。

### 3.3 体检机构与套餐产品

- 平台区分机构主体和分院；用户按机构主体分组浏览，服务能力、地址交通、营业信息、相册、评价、套餐和预约入口均明确标注具体分院。
- 套餐具有稳定业务身份和多个版本；版本包含名称、说明、价格、适用人群、覆盖健康领域和生效状态。
- 普通用户看到当前可售版本及清晰的服务差异，不展示内部表 ID 或数据库式字段表格。
- 机构提交套餐新增、调整、下架和恢复申请；管理员通过后生效，驳回或撤回不影响当前版本。
- 历史预约始终保留下单时的套餐版本、名称和价格事实。

### 3.4 同行体检安排、容量与候补

- 用户按机构、套餐、适用人群、领域与可约日期选择服务。
- 一次安排最多包含五名参与者，可为本人及符合条件的同行家人预约。
- 代预约只代表服务安排，不自动获得对方健康数据查看权限。
- 服务端统一复核所有参与者的日期冲突、套餐适用性和机构名额；任一人不满足条件时整组不创建。
- 日期满额时不能提交预约，可订阅空位提醒；取消或失效释放名额后触发候补通知。
- 通知采用可追踪、可重试且防重复的 outbox 投递机制。

### 3.5 机构履约与健康档案

- 机构工作台按“今日到检—待录入—待复核—已归档”组织任务。
- 受检者、机构、日期和套餐版本由预约带入，机构无需搜索用户健康档案。
- 报告可手工录入或使用识别导入；识别出的低置信度、冲突、无效数值、单位问题和未映射字段必须人工复核。
- 健康档案可包含标准数值指标、文本结论和附件，覆盖现实报告的不同内容形态。
- 归档与预约履约同步完成；归档后内容永久只读，不提供撤下或改写历史的入口。
- 普通用户在健康时间线查看归档结果，趋势与健康 AI 使用同一份获授权事实。
- 机构账号绑定具体分院。本院可创建、编辑、锁定和归档报告；同一机构主体的兄弟分院只可查看已归档报告及附件，草稿和待归档内容不共享，跨机构读取拒绝。
- 跨院详情和附件读取写入访问审计；预约、容量、候补、套餐、邀请码、评价和运营统计始终按分院隔离。

### 3.6 亲友、评论与隐私

- 亲友关系由被添加方控制授权状态；授权只提供脱敏时间线与趋势只读视图。
- 亲友不能代记测量、修改资料、编辑报告或借代预约读取健康内容。
- 亲友响应排除健康身份码、联系方式、病史、内部匹配 ID 等敏感字段。
- 撤权或账号停用后，下一次请求立即重新鉴权。
- 用户完成该机构体检并拥有归档记录后才可评价，管理员负责公开状态审核。

### 3.7 平台治理与账号生命周期

- 管理员总览只显示账号、角色、机构主体、分院、邀请、套餐审核和评论等非健康统计。
- 管理流程为创建机构主体、添加分院、配置分院、签发分院邀请码；不设置总部账号或总部工作台。
- 每个分院可有多个工作人员账号，每个账号只绑定一个分院。
- 邀请码明文仅签发时显示一次，数据库保存哈希；消费与账号创建同步完成。
- 机构软停用；套餐状态通过审核变更；机构工作人员离职不删除历史归档。
- 删除普通用户需明确确认并按数据生命周期规则处理个人健康数据。

### 3.8 健康 AI

- 访客 AI 只做公开导览；机构账号和管理员不使用个人健康上下文。
- 普通问题不查询健康数据；引用记录时需用户逐请求选择并明确同意。
- 可分析内容只能属于本人或已授权亲友，并排除健康身份码和不必要个人资料。
- 服务端计算排序、极值、异常、缺失、来源优先和变化，模型只解释确定性事实。
- 支持 SSE 增量、取消、失败保留和重试；AI 不诊断、不推荐处方药，急症优先提示线下求助。
- 聊天与分析不写业务数据库；私人档案不进入公共知识向量库。

## 4. 技术方案

| 层级 | 选型 |
|---|---|
| 前端 | Vue 3、Vite 6、Vue Router、Pinia、Element Plus、Axios、ECharts 6 |
| 后端 | Flask 3、Flask-SQLAlchemy、Flask-JWT-Extended、Flask-Cors、bcrypt |
| 本地数据库 | SQLite schema v8，`PRAGMA user_version` 管理结构版本 |
| 生产数据库 | openGauss，显式迁移、备份与逐表校验 |
| 文件 | 机构图片、经授权访问的报告附件；识别临时文件按生命周期清理 |
| 报告识别 | 本地 Mock 或华为云 OCR，保守映射并要求人工复核 |
| AI/RAG | SSE、批准的公共知识、本地向量检索、安全规则与确定性健康事实 |
| 测试 | Pytest、Vitest、Vue Test Utils、jsdom、Vite production build |

健康总览、时间线、趋势和机构任务看板由后端提供权限专用 DTO；前端不拼接跨权限数据，也不复用管理员完整对象。

## 5. 数据与权限原则

- 每次请求实时校验角色、账号启用状态、机构绑定和亲友授权。
- 自主测量与机构归档独立建模，但在健康时间线形成统一用户视图。
- “体检数据”只列机构归档报告；个人测量继续进入健康总览、健康时间线和健康趋势。
- 报告以源分院为所有者，兄弟分院获得归档只读能力但不复制数据；所有运营实体仍以分院外键隔离。
- 套餐身份与版本分离；预约和报告保留发生时快照。
- 同行安排、容量扣减、报告归档与套餐版本生效使用事务保证一致性。
- 文件访问通过业务权限验证，不公开整个上传目录。
- 管理员、机构、本人和亲友使用不同 DTO，遵循最小字段原则。
- 健康身份码、报告正文、测量值和个人资料不进入公开日志。

## 6. 数据库升级与演示数据

目标结构为 SQLite schema v8。检查和升级命令：

```powershell
Set-Location .\backend
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py --check-only
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py
```

需要保留原测试账号、覆盖旧业务记录并生成 3.0 综合场景时：

```powershell
.\.venv\Scripts\python.exe .\scripts\reset_v8_demo_data.py --check-only
.\.venv\Scripts\python.exe .\scripts\reset_v8_demo_data.py --apply --yes
```

重置脚本只用于本地合成演示库；若存在非演示个人或机构账号会拒绝执行。生产 openGauss 必须先备份，再进行显式结构迁移与逐表核对，不能使用演示数据重置脚本。

## 7. UI 与非功能需求

- 三角色工作台支持桌面和移动端，具备加载、空状态、错误、403 和 404 页面。
- AI 入口位于顶栏；桌面打开侧栏时主内容按剩余宽度重排，不整体缩放页面。
- 窄屏 AI 使用焦点可达的遮罩层；关怀模式放大字体与关键控件，同时保持导航、卡片和表单不重叠。
- 表格仅用于真正需要精确比较的管理场景；普通用户页面优先使用摘要、卡片、步骤和时间线。
- 密码与邀请码只保存哈希，真实密钥来自被 Git 忽略的环境文件。
- 本地支持 Windows PowerShell，不要求额外数据库服务。
- schema、模型、接口、前端、测试和文档同步更新。

## 8. 当前实现与验收结论

- schema v8、机构主体/分院、跨院归档只读和访问审计，以及健康领域、套餐版本、同行预约、容量、候补通知、文本结果与附件均已实现。
- 日常测量已收拢到健康总览快捷入口，体检记录与自主记录统一进入健康时间线。
- 机构详情、套餐展示和机构工作台已按服务产品与现实履约流程重构。
- 顶栏 AI、响应式重排和重叠问题已纳入前端测试。
- 当前合成快照包含 25 个账号、5 个机构主体、15 家分院、25 个套餐、26 个套餐版本、40 个同行预约组、56 位预约参与者、50 份归档体检报告、120 条自主测量、15 个报告附件和 15 张分院封面。
- 后端 53 项、前端 100 项测试及生产构建通过；浏览器视觉验收由使用者按约定完成。
