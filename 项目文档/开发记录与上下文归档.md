# 开发记录与上下文归档

## 当前状态

- 架构：Vue 3 前端 + Flask 后端。
- 数据库：本地 SQLite。
- 数据文件：`backend/instance/health_system.db`。
- 开发入口：`scripts/start-full-dev.ps1`。
- 本机演示入口：`scripts/start-full-prod.ps1`。
- OCR：本地 Mock 或可选真实 API。
- AI：DeepSeek V4 Flash、固定 FAQ、健康科普安全分流。
- 后端自动化测试：54 项。

## 已完成功能

### 用户认证

- 注册、登录、刷新令牌和当前用户查询。
- bcrypt 密码哈希。
- 图片验证码。
- 前端 Token 自动刷新和路由守卫。

### 机构与套餐

- 机构列表、机构详情和套餐列表。
- 默认种子包含 3 家机构和 15 个套餐。

### 健康档案

- 手动创建、查询和删除档案。
- 指标添加、修改、删除和异常值判断。
- 档案归属人与上传人分离。

### OCR

- 图片和 PDF 上传。
- Mock 与真实 OCR 两种模式。
- 候选映射、未匹配项和人工确认。
- 确认后写入标准指标。

### 亲友授权

- 添加、改名、授权和删除亲友关系。
- 授权后代传亲友档案。
- 授权后查询亲友指标趋势。

### 趋势与评论

- 指标时序点、最新值、最小值和最大值。
- ECharts 折线图。
- 评论提交门槛、管理员审核和公开展示。
- 用户查看和删除自己的评论。

### 管理功能

- 管理员用户列表、修改和删除。
- 管理员评论审核。

### AI 智能助手

- 所有页面挂载可拖动悬浮球；桌面端打开可调宽右侧栏，移动端全屏显示。
- 未登录用户可咨询注册、登录、验证码、OCR 和公开系统功能。
- 登录用户可显式选择同一归属人的最多 5 份本人或已授权亲友档案。
- 后端重新验证档案权限和确认状态，仅将必要的日期、机构、指标值、单位与参考范围发送给模型。
- 健康回答限制为指标解释、基础科普和一般生活建议，不诊断、不推荐处方药。
- 模型用 JSON 返回 `answer/support/emergency`；复杂问题替换为人工客服模板，疑似急症固定提示 120。
- 保留最近 10 轮，更早内容压缩为摘要；后端不建聊天表、不保存消息。
- 浏览器刷新恢复会话，结束对话、登录身份切换或退出登录时清空。

## 本地化决策

项目所有业务数据保存在本地 SQLite。开发模式与 Waitress 演示模式使用同一个数据库文件，不依赖外部数据库或远程主机。

自动化测试使用内存 SQLite，因此测试不会污染实际业务数据。

本地化只改变运行位置，不撤销数据库规范化工作。当前继续保留：

- 3NF 表结构，并完成当前函数依赖范围内的 BCNF 检查。
- 核心候选键和联合唯一约束。
- 22 个命名 CHECK 约束。
- 外键完整性校验。
- 应用层跨表业务规则。

## SQLite 物理结构升级

审计发现旧 `health_system.db` 创建时间早于部分约束模型，因此 `db.create_all()` 无法自动补齐 25 个命名约束。

已新增 `backend/scripts/upgrade_local_database.py` 并完成升级：

- 迁移前扫描现有数据，没有发现约束违规。
- 自动生成原数据库时间戳备份。
- 根据最新 SQLAlchemy 模型重建表。
- 按外键依赖顺序复制全部业务数据。
- 升级前后所有表行数一致。
- 缺失约束为 0，外键违规为 0，完整性检查为 `ok`。

同时在 `backend/app/extensions.py` 中为 SQLite 连接开启
`PRAGMA foreign_keys=ON` 和 5 秒锁等待。

## AI 架构决策

- 选用 `deepseek-v4-flash` 非思考模式：面向高频短问答，复杂医疗问题不交给模型深度推理。
- 登录前优先匹配本地 FAQ，未命中才调用模型，减少费用并保证常见系统答案稳定。
- API Key 只保存在被 Git 忽略的 `backend/.env`，由 Flask 代理请求，前端无法读取。
- 不新增聊天数据表。上下文由浏览器发送最近 10 轮和摘要，后端保持无状态，符合“结束后不保存历史”的需求。
- 使用本地急症关键词规则作为第一道安全网，再由模型 JSON 分类；`support` 和 `emergency` 的最终文本由后端模板控制。

## 后续开发约定

- 数据模型变更必须同步测试和数据库设计文档。
- 新增配置先写入 `.env.example`，真实密钥只放 `.env`。
- 开发时优先使用 `start-full-dev.ps1`。
- 修改数据前备份 `backend/instance/health_system.db`。
- 模型约束变化后运行 `scripts/upgrade_local_database.py`，不能只依赖 `db.create_all()`。
- 上传附件需要与数据库一并备份。
- AI 提示词、安全边界、FAQ 或接口结构变化时同步更新前后端 README、需求文档和测试报告。
- 禁止把 DeepSeek API Key、用户对话或健康指标写入源码、示例配置、日志和提交记录。
