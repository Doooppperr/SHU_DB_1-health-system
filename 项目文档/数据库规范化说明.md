# 康康健健 HealthDoc 数据库规范化说明

> 文档状态：SQLite schema v3 当前规范化与完整性说明，更新于 2026-07-16。

## 1. 规范化目标

账号、机构、套餐、邀请码、图片、指标字典、自测、体检登记、机构报告、报告指标、亲友和评论分别建模。当前关系以 3NF 为最低目标，并在已识别函数依赖范围内满足 BCNF。

- `self_measurements`：用户自主记录的时间点测量；
- `exam_registrations`：用户表达的体检机构/日期意图；
- `institution_reports` + `report_indicators`：机构锁定和发布的正式事实。

三种实体分别承载不同责任主体、生命周期和可信级别。

## 2. 第一至第三范式

### 第一范式（1NF）

- 报告指标以 `report_indicators` 多行保存，不为不同指标动态增加列。
- 日常测量以一条测量一行保存，同日多次记录不塞进 JSON 数组。
- 机构套餐、邀请码和图片分别建表，不在机构表保存重复多值字段。
- 亲友与评论使用独立关系，不把关系列表写入用户列。

`indicator_dicts.aliases` 与 `institution_reports.ocr_diagnostics` 是有意保留的 JSON 文档值：别名集合由映射服务整体读取，不参与连接/候选键；OCR 诊断是外部解析快照，不是正式健康事实。

### 第二范式（2NF）

- `report_indicators` 的值、异常和来源依赖完整候选键 `(report_id, indicator_dict_id)`。
- 机构套餐属性依赖 `(institution_id, name)` 对应的套餐实体，而不是只依赖机构。
- 机构图片位置依赖 `(institution_id, sort_order)`，图片存储键另有唯一性。
- 亲友授权依赖完整用户对 `(user_id, friend_user_id)`。

### 第三范式（3NF）

- 自测和报告指标只引用 `indicator_dict_id`，单位、参考范围与临床意义留在指标字典。
- 账号通过 `managed_institution_id` 引用机构，不复制机构名称。
- 登记和报告通过机构/套餐外键，不复制套餐描述与价格。
- 评论引用用户和机构，不重复保存用户名和机构名。
- 报告创建者用户名快照是有意的历史事实，不用于推导当前账号属性；它在账号删除后仍需保持创建时显示值。
- `report{id}` 为主键派生展示值，不重复落库。

## 3. 候选键

| 表 | 候选键 |
|---|---|
| `users` | `username`、非空 `email`、非空 `health_id` |
| `institutions` | `(name, branch_name)` |
| `packages` | `(institution_id, name)` |
| `institution_invites` | `institution_id`、`code_hash` |
| `institution_images` | `(institution_id, sort_order)`、`storage_key` |
| `indicator_categories` | `name` |
| `indicator_dicts` | `code`、`(category_id, name)` |
| `friend_relations` | `(user_id, friend_user_id)` |
| `exam_registrations` | 对 `status <> cancelled` 的 `(user_id, exam_date)` 部分候选键；`matched_report_id` 非空时唯一 |
| `institution_reports` | 对活动状态的 `(institution_id, subject_health_id, exam_date)` 部分候选键；`exam_registration_id` 非空时唯一 |
| `report_indicators` | `(report_id, indicator_dict_id)` |

`users.managed_institution_id` 不是候选键，因为业务允许一个机构绑定多个账号。

## 4. BCNF 检查

- 用户主键、用户名、非空邮箱和非空健康身份码都决定同一用户属性。
- 机构主键或 `(name, branch_name)` 决定机构属性。
- 邀请码的机构候选键和哈希候选键决定当前邀请码状态。
- 套餐、图片、指标字典、亲友关系和报告指标由各自候选键决定属性。
- 活动登记的 `(user_id, exam_date)` 决定当前登记；取消行不参与该部分依赖。
- 活动报告的 `(institution_id, subject_health_id, exam_date)` 决定当前报告；撤下/过期行不参与该部分依赖。

“姓名 + 健康身份码 + 日期 + 机构匹配”“发布报告后允许评论”“机构报告同日优先于自测”是跨表业务规则，不通过复制用户、评论资格或最终趋势值实现，因此不会在关系中引入新的传递依赖。

## 5. 有意反规范化与派生值

### 创建者用户名快照

`institution_reports.created_by_username_snapshot` 与 `created_by_user_id` 同时存在。这不是为了缓存当前用户名，而是保存报告创建时的历史显示事实。机构账号可以被删除，外键置空后快照仍有业务意义。

### 受检者姓名和健康身份快照

报告在匹配前必须保存机构录入的 `subject_name_snapshot` 和 `subject_health_id`。它们是机构提交的原始匹配声明，不是从当前用户资料自动派生。匹配成功后仍保留，用于证明报告当时依据；用户视图和亲友 DTO会隐藏敏感字段。

### OCR 诊断 JSON

`ocr_diagnostics` 保存一次解析的候选、未匹配、过滤与完整性诊断。正式指标仍规范化到 `report_indicators`；诊断不参与趋势、异常或匹配。

### 双向一对一引用

登记与报告各保存一个唯一外键，属于性能和导航便利的受控冗余。匹配服务必须在同一事务写入两端；测试验证不会形成一对多或单边关联。

## 6. 数据完整性

### 数据库约束

- 角色限制为 `user/institution_admin/admin`。
- 机构账号必须绑定机构；普通用户和管理员不能绑定。
- 普通用户必须有健康身份码；两类管理员不能有。
- 健康身份码、用户名和非空邮箱唯一。
- 性别、邀请状态、登记状态、报告状态、指标值类型、报告指标来源均有检查约束。
- 自测值、套餐价格非负；评论评分为 1–5。
- 同机构最多 8 个排序位置，每个位置唯一。
- 同用户同日最多一个活动登记；同机构/健康身份/日期最多一个活动报告。
- 一个登记最多匹配一个报告，一个报告最多绑定一个登记。

### 服务端事务约束

- 自测指标必须在六项白名单且只能操作本人数据。
- 登记或报告套餐必须属于当前机构。
- 邀请码必须当前有效，消费与机构账号创建原子提交。
- 报告只允许草稿写入；锁定、提交、匹配、撤下按状态条件更新。
- 匹配四项全部相等，且同时校验双方未被占用、账号/机构有效和 60 天期限。
- 报告锁定成功才删除临时文件；事务失败时新文件补偿清理。
- 普通用户删除需显式确认，机构账号删除保留报告快照。

## 7. 健康数据与权限边界

- 本人完整个人资料、亲友身份 DTO、机构报告 DTO和管理员账号 DTO分开序列化。
- 亲友只读投影不包含个人资料、联系方式、健康身份或内部匹配字段。
- 机构账号只能按数据库绑定机构查询和修改报告，不能提交其他机构 ID。
- 管理员 dashboard 和账号接口不查询或返回健康表内容。
- AI 只读取普通用户本人或授权亲友的 `published` 报告与每日有效趋势事实。
- 草稿临时原文件不公开，锁定后删除；机构图片公开仍需数据库登记。
- AI 会话、模型回复和最终趋势序列不落库；趋势每次从有效来源确定性计算。

因此三工作台不会产生各自一份重复健康档案，也不会用缓存表存放可能失效的亲友或管理员投影。

## 8. 趋势计算为何不落表

每日有效趋势依赖报告当前状态和同日自测顺序：

1. 查找该日该指标的 `published` 机构报告指标；
2. 存在时选择机构值；
3. 不存在时选择 `measured_at` 最晚的自测；
4. 报告撤下后下一次查询自然回退到自测。

如果把最终值持久化，需要在报告发布、撤下、自测新增/修改/删除时维护多处缓存，容易产生更新异常。因此当前选择查询时确定性计算，以一致性优先。

## 9. 删除与生命周期

- 机构、套餐软停用，保留历史外键。
- 机构账号物理删除，报告创建者外键 `SET NULL`，用户名快照保留。
- 普通用户删除会级联自测、登记、匹配报告和报告指标；未匹配且只含机构输入的报告不因无关用户删除而受影响。
- 报告撤下保留行与指标，但所有用户读取统一过滤。
- 未匹配报告到期后标记/删除，避免无限保存无法归属的健康信息。
- 图片和草稿文件与数据库事务配合清理；孤儿扫描只在受限上传目录运行。

## 10. 物理落库结果与结论

本地正式数据库当前为从空库生成的 schema v3 合成演示快照；旧库升级路径仍由自动化验证：

- `user_version=3`；
- `integrity_check=ok`；
- 外键违规 0；
- 旧库升级时管理员主键与密码哈希保留；
- 重建前数据库完整保存在本机时间戳备份；
- 新表、列、命名约束和部分唯一索引通过测试；
- 当前 5 个普通用户均有丰富时间线，覆盖发布、等待匹配、撤下、同日多测、趋势回退和异常标记。

当前模型满足 3NF，并在已识别函数依赖范围内满足 BCNF。有意快照、OCR JSON 和双向引用都有明确语义、事务维护与权限边界，不是无约束的数据重复。
