# 康康健健 HealthDoc 数据库规范化说明

> 文档状态：SQLite/openGauss schema v7 规范化与完整性说明，更新于 2026-07-20。

## 1. 规范化范围

schema v7 以 3NF 为最低目标，并在已识别函数依赖范围内满足 BCNF。账号、机构、套餐版本、健康领域、同行安排、个人预约、状态事件、容量、候补、健康档案、文本结果、附件、亲友授权和评论分别建模，数据库结构服务于现实业务流程而不是页面表格。

## 2. 范式检查

### 1NF

- 一次自主测量、一项报告指标、一段文本结论、一个附件和一个预约事件分别成行。
- 同行体检的每名参与者使用独立 `appointments` 记录，不把参与者数组塞入预约主表。
- 套餐覆盖的健康领域使用 `package_version_domains` 关联，不使用逗号字符串。
- 只有识别诊断、审核快照、联系方式快照和通知载荷等不可分审计事实使用 JSON。

### 2NF

- 报告指标依赖完整键 `(report_id, indicator_dict_id)`。
- 套餐领域关联依赖完整键 `(package_version_id, health_domain_id)`。
- 同行参与者、候补参与者和事件均依赖所属业务对象，不把组级属性重复到参与者明细。
- 亲友授权依赖完整用户对 `(user_id, friend_user_id)`。

### 3NF

- 指标单位、范围和领域归属由指标字典及领域关联维护，测量和报告只引用字典 ID。
- 机构资料不重复写入套餐、预约或报告；历史页面需要的内容使用发生时快照表达。
- 套餐当前身份与历史版本分离，价格、适用人群和领域属于具体版本。
- 预约状态变化进入事件表，避免用多个页面字段重复推导历史。
- 报告展示编号由主键派生，不重复落库。

## 3. 主要候选键与唯一性

| 表 | 候选键或唯一规则 |
|---|---|
| `users` | `username`、非空 `email`、非空 `health_id` |

其中 `username` 和普通用户的 `health_id` 承担账号/健康身份唯一性；`email` 是注册必填的通知渠道，不承担账号识别，因此允许家庭成员或演示角色共享同一邮箱。
| `institutions` | `(name, branch_name)` |
| `packages` | 机构内稳定业务标识/名称唯一规则 |
| `package_versions` | `(package_id, version_number)` |
| `health_domains` | 领域编码 |
| `indicator_domain_links` | `(indicator_dict_id, health_domain_id)` |
| `package_version_domains` | `(package_version_id, health_domain_id)` |
| `appointments` | 有效服务日期上的用户冲突约束，组内参与者唯一 |
| `appointment_capacity_slots` | `(institution_id, appointment_date)` |
| `waitlist_subscriptions` | 用户、机构、套餐版本、日期范围的有效订阅规则 |
| `institution_reports` | 预约唯一；机构、受检者、体检日期的归档唯一规则 |
| `report_indicators` | `(report_id, indicator_dict_id)` |
| `friend_relations` | `(user_id, friend_user_id)` |

一个机构允许多个工作人员账号，因此 `managed_institution_id` 不是 `users` 的候选键。

## 4. 有意保留的历史快照

- 预约保存下单时的受检者、联系方式、套餐版本、名称和价格。
- 报告保存归档时的受检者及创建者展示信息。
- 套餐审核保存变更前后内容，确保管理员决策可追溯。
- 报告识别诊断保存候选、置信度和需人工复核项。
- 通知 outbox 保存当次投递载荷，确保重试内容不随当前实体变化。

这些值表达“当时发生了什么”，不能从当前机构、账号或套餐状态无损重算，因此不属于冗余更新异常。

## 5. 完整性与事务规则

### 数据库约束

- 账号角色、预约状态、报告状态、审核状态和通知状态使用受限枚举。
- 价格与测量数值满足业务范围，评论评分为 1–5。
- 外键明确级联、限制或置空语义；历史归档不会因机构工作人员离职而丢失。
- 套餐版本号、日期容量、报告指标、领域关联和亲友关系具有唯一约束。

### 服务端事务

- 同行安排先校验全部参与者、日期冲突、套餐适用性和容量，再整体写入。
- 预约取消/失效与容量释放同步完成，满额释放后生成候补通知事件。
- 报告归档与预约履约同步完成；归档内容不可再编辑。
- 套餐审核通过与新版本生效同步完成，驳回或撤回不修改正式服务产品。
- 邀请码消费与机构账号创建同步完成。
- outbox 投递失败只保留重试状态，不破坏已提交的业务事实。

## 6. 查询投影而非重复落表

- 健康总览由最近测量、归档报告和预约进度聚合生成。
- 健康时间线按 `record_type=all|exam|self` 查询统一投影，不复制一份“时间线表”。
- 健康趋势按领域和指标实时计算；同日机构结果优先，缺少时使用当天最后一次自主测量。
- 亲友视图使用脱敏 DTO，不复制亲友健康数据。

这样可以避免报告归档、自主测量修改或授权撤销后维护多份派生数据。

## 7. 文件与隐私边界

- 报告附件以 `report_assets` 元数据关联归档记录，访问仍需业务鉴权。
- 机构图片与健康附件分开管理，不能通过公共上传目录猜测健康文件地址。
- 管理员运营统计不读取个人测量、报告正文或健康档案。
- AI 只读取本人或已授权亲友的获准记录，聊天内容不写入业务表。

## 8. 迁移、重置与当前规模

目标结构为 `PRAGMA user_version=7`。结构升级使用：

```powershell
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py --check-only
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py
```

本地第三轮演示数据使用：

```powershell
.\.venv\Scripts\python.exe .\scripts\reset_v7_demo_data.py --check-only
.\.venv\Scripts\python.exe .\scripts\reset_v7_demo_data.py --apply --yes
```

当前合成快照包含 13 个账号、3 家机构、9 个套餐、10 个套餐版本、15 个同行预约组、17 个预约、9 条归档健康数据/报告、70 条自主测量、3 条候补订阅、9 条文本结果、3 个报告附件和 3 条评论。重置脚本保留测试账号，且只允许作用于经过识别的本地演示环境。
